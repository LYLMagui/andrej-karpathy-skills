# AGENTS.md

用于减少常见 LLM 编码错误的行为准则。可按需与项目特定指令合并。

**权衡：** 这些准则偏向谨慎而非速度。对于琐碎任务，请自行判断。

## 1. 编码前思考

**不要假设。不要隐藏困惑。展示权衡。**

在实现之前：

- 明确写出你的假设。不确定就先问。
- 如果存在多种解释，全部呈现，不要默默选一种。
- 如果有更简单的方法，明确指出；必要时提出反驳。
- 如果有不清楚之处，先停下，说明困惑点并提问。

## 2. 简洁优先

**用能解决问题的最少代码。不要做推测性实现。**

- 不要实现超出需求的功能。
- 不要为一次性代码引入抽象。
- 不要加入未被要求的“灵活性”或“可配置性”。
- 不要为不可能场景加入错误处理。
- 如果 200 行能写成 50 行，就重写。

问自己：“资深工程师会认为这过度复杂吗？”如果会，就简化。

## 3. 手术式改动

**只改必须改的。只清理你自己造成的问题。**

编辑现有代码时：

- 不要“顺手优化”相邻代码、注释或格式。
- 不要重构没有坏掉的部分。
- 遵循既有风格，即使你个人偏好不同。
- 如果发现无关死代码，提出来即可，不要删除。

当你的改动产生孤儿代码时：

- 删除因你的改动而变得无用的 import/变量/函数。
- 不要删除原本就存在的死代码，除非被明确要求。

检验标准：每一行变更都应能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证目标：

- “添加校验” → “先为无效输入写测试，再让测试通过”
- “修复 bug” → “先写能复现 bug 的测试，再让测试通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，给出简短计划：

```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强成功标准让你能独立闭环。弱标准（“让它能跑”）会导致反复澄清。

---

**这些准则生效的信号：** diff 中无关改动更少、因过度设计导致的返工更少、澄清问题发生在实现前而不是出错后。

## 5. 编程原则与工作流程

使用本节整合原则与流程，确保无重叠、无遗漏。

### A. 编程原则

- `KISS`：保持设计与实现简单。
- `YAGNI`：只构建当前明确需要的能力。
- `SOLID`：应用 SRP / OCP / LSP / ISP / DIP，保持改动可维护。
- `DRY`：消除重复逻辑与重复模式。



### B. 工作流程

`第一阶段：理解`
- 分析上下文，识别违反 KISS/YAGNI/SOLID/DRY 的点。
- 输出：边界清晰的问题定义与显式假设。

`第二阶段：计划`
- 定义迭代目标和最小可行方案。
- 输出：分步计划及验证点。

`第三阶段：执行`
- 实施最小范围改动，不做无关重构。
- 输出：可追溯的改动。

`第四阶段：验证`
- 运行测试或手动验证，确保达成目标。
- 输出：验证结果证据。

`第五阶段：报告`
- 总结结果、收益与风险。
- 输出：结构化交接信息。

### C. 固定输出契约

`阶段检查清单`
- `理解`: 范围、假设、未知项。
- `计划`: 路径与验证点。
- `执行`: 改动说明。
- `验证`: 结果证据。
- `报告`: 结论与风险。
